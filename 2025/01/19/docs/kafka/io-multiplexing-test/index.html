C-IO multiplexing-test

FD는 자원을 식별하는 숫자

공통점

  • 두 방식 모두 감시할 파일 디스크립터 수가 많아질수록 성능 저하 발생.
  • 선형적으로 모든 FD를 확인해야 하므로 비효율적.
  • 대규모 시스템에서는 성능이 병목이 될 수 있음.

select vs poll 차이점

특징 select poll
API 방식 fd_set 사용 pollfd 배열 사용
FD 개수 제한 제한 있음 (FD_SETSIZE, 보통 1024) 제한 없음
이벤트 종류 읽기, 쓰기, 예외만 처리 가능 다양한 이벤트 처리 가능 (POLLIN, POLLOUT 등)
호출 후 결과 fd_set 수정, 원본 복원 필요 pollfd 배열에 결과 저장
성능 많은 FD에서 비효율적 많은 FD에서 비효율적(그러나 약간 더 유연함)
포팅 가능성 더 널리 지원됨 현대 시스템에서 많이 사용

운영 체제는 열린 자원을 FD 테이블에 저장하고 관리합니다. 이를 통해:

  • 자원의 상태(열림/닫힘)를 추적.
  • 프로세스가 FD를 통해 자원을 사용할 수 있도록 인터페이스 제공.
  • 자원이 사용 중일 때 중복 사용 방지.

select

  • 매번 호출 시 모든 파일 디스크립터를 선형적으로 확인.

  • poll

  • int poll(struct pollfd *fds, nfds_t nfds, int timeout);

I/O 멀티플렉싱

epoll (리눅스), kqueue (mac_)

  • epoll: 리눅스에서 select/poll의 성능 문제를 해결하기 위해 도입된 비차단 I/O 멀티플렉싱.
    • 장점: 대규모 FD를 효율적으로 관리, 이벤트 기반으로 동작.
    • 단점: 특정 운영 체제(Linux)에서만 동작.
  • kqueue: BSD 및 macOS에서 제공되는 유사한 고성능 인터페이스.

Kafka의 poll과 일반 poll의 관계?

Kafka의 poll은 이름만 같을 뿐, I/O 멀티플렉싱의 poll과는 본질적으로 다릅니다. Kafka poll은 네트워크 통신을 추상화한 고수준 API로, Kafka 브로커에서 데이터를 가져오는 동작을 관리하는 메서드인 반면, 일반적인 poll은 운영 체제 수준에서 파일 디스크립터의 I/O 이벤트를 감시하는 저수준 함수이다.

FD와 NIO 관점에서의 공통점

1. 파일 디스크립터(FD)의 역할

  • FD는 운영 체제 수준에서 자원을 식별하는 추상화된 숫자이다.
    • C에서는 파일 디스크립터가 직접적으로 사용됨
    • Java NIO에서도 FD는 내부적으로 사용되며, 이를 추상화한 객체로 SelectableChannel이 제공됨
    • Kafka의 poll은 FD를 직접 다루지 않지만, 네트워크 소켓을 기반으로 통신하므로 FD를 간접적으로 활용

2. 공통적인 작동 원리

  • FD를 기반으로 이벤트 감지:
    • select와 poll은 파일 디스크립터를 감시하여 I/O 이벤트(읽기, 쓰기 등)를 감지
    • Java NIO의 Selector는 FD 기반으로 동작하는 멀티플렉싱 기법(epoll 등)을 내부적으로 활용
    • Kafka poll도 브로커와의 통신에서 내부적으로 소켓과 FD를 이용하며, 이는 Java NIO로 구현

Kafka poll과 C의 poll, select의 FD 관점 주요 차이

특징 Kafka poll C poll C select
FD 관리 내부적으로 NIO Selector가 FD를 관리 사용자가 직접 FD를 등록 사용자가 직접 FD를 등록
FD 사용 방식 소켓 기반으로 네트워크 통신 파일/소켓 FD에서 직접 이벤트 감지 파일/소켓 FD에서 직접 이벤트 감지
FD 제한 제한 없음 (NIO와 epoll 기반) 제한 없음 FD 개수 제한 (FD_SETSIZE, 보통 1024)
초점 Kafka 메시지 처리 (FetchRequest/응답) 비동기 I/O 처리 비동기 I/O 처리
추상화 수준 고수준 API 제공 저수준 API 저수준 API
