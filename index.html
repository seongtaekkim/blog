consumer-async

요청흐름 다이어그램

Online FlowChart & Diagrams Editor - Mermaid Live Editor

[컨슈머 요청 흐름]

1. Consumer.commitAsync(callback)

  • 역할:
    • 사용자 애플리케이션에서 커밋 요청을 시작
    • 오프셋 커밋 작업이 비동기로 수행되며, 커밋 결과를 처리할 콜백(callback)을 제공
  • 발생 위치: 사용자 코드.
  • 결과: 내부적으로 ConsumerCoordinator.commitOffsetsAsync가 호출됨

2. ConsumerCoordinator.commitOffsetsAsync

  • 역할:
    • 비동기 방식으로 오프셋 커밋을 요청한다
    • 내부적으로 오프셋 데이터를 준비하고, 브로커로 전송할 준비시작
  • 발생 위치: ConsumerCoordinator 클래스.
  • 결과: doCommitOffsetsAsync 호출.

3. ConsumerCoordinator.doCommitOffsetsAsync

  • 역할:
    • 커밋할 오프셋 데이터를 검증하고, 네트워크 요청 객체를 생성
    • 브로커로 요청 전송을 준비함
  • 발생 위치: ConsumerCoordinator 클래스.
  • 결과: sendOffsetCommitRequest 호출.

4. ConsumerCoordinator.sendOffsetCommitRequest

  • 역할:
    • OffsetCommitRequest 객체를 생성
    • 요청과 함께 응답을 처리할 핸들러(offsetCommitResponseHandler)를 등록한다
  • 발생 위치: ConsumerCoordinator 클래스.
  • 결과: Consumer.client.send 호출.

5. Consumer.client.send

  • 역할:
    • 네트워크 요청(OffsetCommitRequest)을 Kafka 브로커로 전송한다
    • 응답 핸들러가 네트워크 요청과 함께 등록된다
  • 발생 위치: ConsumerNetworkClient 클래스.
  • 결과: 브로커에 요청 전송.

[브로커 처리 흐름]

6. KafkaApis.handleOffsetCommitRequest

  • 역할:
    • 브로커에서 요청을 수신한다
    • 요청의 유효성을 검사하고, 그룹 코디네이터로 전달
  • 발생 위치: 브로커의 KafkaApis 클래스.
  • 결과: GroupCoordinatorAdapter.handleCommitOffsets 호출.

7. GroupCoordinatorAdapter.handleCommitOffsets

  • 역할:
    • 그룹 코디네이터(GroupCoordinator)의 오프셋 커밋 로직을 호출한다
    • 요청된 오프셋 데이터를 그룹 관리 로직으로 전달한다
  • 발생 위치: 브로커의 GroupCoordinatorAdapter 클래스.
  • 결과: GroupCoordinator.handleCommitOffsets 호출.

8. GroupCoordinator.handleCommitOffsets

  • 역할:
    • 그룹 관련 데이터를 검증하고, 오프셋 커밋 요청을 처리한다
    • _consumer_offsets 토픽으로 오프셋 데이터를 저장하기 위해 준비한다
  • 발생 위치: 브로커의 GroupCoordinator 클래스.
  • 결과: GroupCoordinator.doCommitOffsets 호출.

9. GroupCoordinator.doCommitOffsets

  • 역할:
    • _consumer_offsets 토픽에 기록할 데이터를 준비한다
    • 데이터를 저장하기 위해 ReplicaManager.appendRecords를 호출한다
  • 발생 위치: 브로커의 GroupCoordinator 클래스.
  • 결과: ReplicaManager.appendRecords 호출.

10. ReplicaManager.appendRecords

  • 역할:
    • _consumer_offsets 토픽의 리더 파티션에 데이터를 저장한다
    • ISR(In-Sync Replica)에 복제를 수행함
  • 발생 위치: 브로커의 ReplicaManager 클래스.
  • 결과: 저장 완료 후 응답 콜백 호출.

11. responseCallback 호출 (오프셋 저장 결과)

  • 역할:
    • _consumer_offsets에 오프셋이 저장된 결과를 처리한다
    • 저장 성공 또는 실패 여부를 상위 호출자로 전달한다
  • 발생 위치: GroupCoordinator.doCommitOffsets.
  • 결과: GroupCoordinator 계층의 콜백 호출.

12~14. GroupCoordinator 계층의 responseCallback 처리

  • 역할:
    • 응답 콜백을 통해 저장 성공 여부를 상위 호출자에게 전달한다
    • 최종적으로 KafkaApis에 결과를 반환한다
  • 발생 위치: GroupCoordinator 및 GroupCoordinatorAdapter.
  • 결과: KafkaApis.handleOffsetCommitRequest로 결과 전달.

15. KafkaApis.handleOffsetCommitRequest에서 OffsetCommitResponse 생성

  • 역할:
    • 저장 결과를 기반으로 응답 객체(OffsetCommitResponse)를 생성한다
    • 클라이언트로 반환할 준비를 완료
  • 발생 위치: 브로커의 KafkaApis.
  • 결과: KafkaApis.sendResponseMaybeThrottle 호출.

16. KafkaApis.sendResponseMaybeThrottle

  • 역할:
    • 클라이언트 응답 전 스로틀링을 적용할지 확인한다
    • 필요 시 응답을 지연시킨다
  • 발생 위치: 브로커의 KafkaApis.
  • 결과: KafkaApis.sendResponse 호출.

17. RequestChannel.sendResponse

  • 역할:
    • 브로커가 응답 데이터를 클라이언트로 전송한다
  • 발생 위치: 브로커의 RequestChannel.
  • 결과: 클라이언트로 응답 전송 완료.

[컨슈머 응답 처리]

18. Consumer.client.poll()로 응답 수신

  • 역할:
    • 컨슈머가 브로커의 응답을 수신한다
    • 등록된 핸들러를 실행합니다.
  • 발생 위치: ConsumerNetworkClient 클래스.

19. offsetCommitResponseHandler 호출

  • 역할:
    • 응답 데이터를 처리하고, 성공/실패를 판별한다
  • 발생 위치: ConsumerCoordinator.
  • 결과: 사용자 콜백 호출.

20. 사용자 콜백(callback) 호출

  • 역할:
    • 사용자가 commitAsync 호출 시 등록한 콜백을 호출한다
    • 커밋 성공/실패 결과를 애플리케이션에 전달한다
  • 발생 위치: 사용자 코드.

컨슈머에서 poll을 이용해 데이터를 받아온다고 해보자

KafkaConsumer 인스턴스를 생성하면

실제로 AsyncKafkaConsumer 인스턴스가 생성되면서 ConsumerNetworkThread 스레드가 생성,시작된다.

이후 poll 을 호출하면 awaitNotEmpty 메서드에 의해 시간인자만큼 해당스레드에 락이 걸린다.

ConsumerNetworkThread 는 networkClient 의 poll을 호출하여 NIO 동작을 하며 요청데이터를 수신한다.

즉, 사용자는 poll이 블로킹하는 것으로 보이지만, 내부적으로 생성된 스레드에 의해 비동기 동작함을 알 수 있다.

C-IO multiplexing-test

FD는 자원을 식별하는 숫자

공통점

  • 두 방식 모두 감시할 파일 디스크립터 수가 많아질수록 성능 저하 발생.
  • 선형적으로 모든 FD를 확인해야 하므로 비효율적.
  • 대규모 시스템에서는 성능이 병목이 될 수 있음.

select vs poll 차이점

특징 select poll
API 방식 fd_set 사용 pollfd 배열 사용
FD 개수 제한 제한 있음 (FD_SETSIZE, 보통 1024) 제한 없음
이벤트 종류 읽기, 쓰기, 예외만 처리 가능 다양한 이벤트 처리 가능 (POLLIN, POLLOUT 등)
호출 후 결과 fd_set 수정, 원본 복원 필요 pollfd 배열에 결과 저장
성능 많은 FD에서 비효율적 많은 FD에서 비효율적(그러나 약간 더 유연함)
포팅 가능성 더 널리 지원됨 현대 시스템에서 많이 사용

운영 체제는 열린 자원을 FD 테이블에 저장하고 관리합니다. 이를 통해:

  • 자원의 상태(열림/닫힘)를 추적.
  • 프로세스가 FD를 통해 자원을 사용할 수 있도록 인터페이스 제공.
  • 자원이 사용 중일 때 중복 사용 방지.

select

  • 매번 호출 시 모든 파일 디스크립터를 선형적으로 확인.

  • poll

  • int poll(struct pollfd *fds, nfds_t nfds, int timeout);

I/O 멀티플렉싱

epoll (리눅스), kqueue (mac_)

  • epoll: 리눅스에서 select/poll의 성능 문제를 해결하기 위해 도입된 비차단 I/O 멀티플렉싱.
    • 장점: 대규모 FD를 효율적으로 관리, 이벤트 기반으로 동작.
    • 단점: 특정 운영 체제(Linux)에서만 동작.
  • kqueue: BSD 및 macOS에서 제공되는 유사한 고성능 인터페이스.

Kafka의 poll과 일반 poll의 관계?

Kafka의 poll은 이름만 같을 뿐, I/O 멀티플렉싱의 poll과는 본질적으로 다릅니다. Kafka poll은 네트워크 통신을 추상화한 고수준 API로, Kafka 브로커에서 데이터를 가져오는 동작을 관리하는 메서드인 반면, 일반적인 poll은 운영 체제 수준에서 파일 디스크립터의 I/O 이벤트를 감시하는 저수준 함수이다.

FD와 NIO 관점에서의 공통점

1. 파일 디스크립터(FD)의 역할

  • FD는 운영 체제 수준에서 자원을 식별하는 추상화된 숫자이다.
    • C에서는 파일 디스크립터가 직접적으로 사용됨
    • Java NIO에서도 FD는 내부적으로 사용되며, 이를 추상화한 객체로 SelectableChannel이 제공됨
    • Kafka의 poll은 FD를 직접 다루지 않지만, 네트워크 소켓을 기반으로 통신하므로 FD를 간접적으로 활용

2. 공통적인 작동 원리

  • FD를 기반으로 이벤트 감지:
    • select와 poll은 파일 디스크립터를 감시하여 I/O 이벤트(읽기, 쓰기 등)를 감지
    • Java NIO의 Selector는 FD 기반으로 동작하는 멀티플렉싱 기법(epoll 등)을 내부적으로 활용
    • Kafka poll도 브로커와의 통신에서 내부적으로 소켓과 FD를 이용하며, 이는 Java NIO로 구현

Kafka poll과 C의 poll, select의 FD 관점 주요 차이

특징 Kafka poll C poll C select

ToxiProxy는 네트워크 조건을 시뮬레이션하기 위한 TCP 프록시이다

  1. Toxic 종류:

Direction(방향) 설정:

파라메터

  • toxicName: Toxic의 고유 식별자

  • 네트워크 불안정성 테스트

트랜잭션 테스트 (네트워크 지연)

1. 트랜잭션 설정

  • 트랜잭션 ID를 지정하여 트랜잭션 기능 활성화
  • 멱등성을 활성화하여 메시지 중복 전송 방지
  • 트랜잭션 타임아웃은 35초로 설정

2. 트랜잭션 초기화

  • Producer가 트랜잭션 코디네이터와 연결을 설정
  • PID(Producer ID)를 할당받음
  • 이전 트랜잭션들의 상태를 확인하고 정리

3. 트랜잭션 시작

  • 새로운 트랜잭션 시작
  • 이후의 모든 메시지 전송은 이 트랜잭션에 포함됨

4. 메시지 전송 시도와 실패

5. 실패 처리와 롤백

6. 트랜잭션 격리 수준

  • read_committed로 설정된 컨슈머는 롤백된 메시지를 볼 수 없음
  • 실패한 트랜잭션의 메시지는 완전히 제거됨

이 테스트는 네트워크 장애 상황에서 Kafka 트랜잭션의 원자성(Atomicity)을 검증한다

  • 모든 메시지가 성공적으로 전송되거나
  • 아무 메시지도 전송되지 않아야 함 (all or nothing)

1. 파드(Pod) ↔ 파드간 통신

pod1, pod2 가상인터페이스 확인.

Pod 간 통신

  • 라우팅 정보에 169.254.1.1을 디폴트 게이트웨이 주소로 사용
  • calice 인터페이스(파드에 연결된 가상 인터페이스)에 proxy_arp 파일 (활성화)

ARP 정보를 어떻게 가져올까? - proxy_arp 기능을 이용

  • 파드1에서 게이트웨이의 IP인 169.254.1.1 의 MAC 주소를 알기 위해서 ARP Request 를 보낸다
  • 이때 veth 연결된 calice#~ 에 proxy arp 설정이 되어 있고, 자신의 mac 주소를 알려주고, 이후 정상 통신됨

  • 각 노드에 파드 네트워크 대역은 Bird 에 의해서 BGP 로 광고 전파/전달 되며, Felix 에 의해서 호스트의 라우팅 테이블에 자동으로 추가 및 삭제 됩니다

Container Layer Stack

  • 컨테이너 설치 시 이미지가 어떤 방식, 구조로 저장되는지 알아보자

  • 도커 컨테이너 , 이미지 삭제

  • docker pull nginx:latest
  • image id 확인
  • layer id 확인 (image id 조회)
  • 도커는 다운로드한 이미지를 로컬에 저장하고 관리하기 위해 **”레이어 DB”**를 사용한다.
레이어디비 항목 내용
db id 로컬에서 레이어식별을 위한 레이어디비 id
이미지를 pull 받을 때 레이어 별로 생성됨. (받을 때마다 달라짐)
cache id 레이어가 저장된 로컬경로 id (경로를 식별)
- /var/lib/docker/overlay2/{cache-id}
diff layer id (레이어를 식별)
parent 부모레이어를 가리키는 포인터 (db id) 어떤 레이어 위로 올라갈지 지정

db id

  • tree -L 2 sha256 -I “tar-*|size” 에서 -I 는 제외항목을 작성

cache-id : db-id

  • cache-id 리스트
    • 포맷: “cache-id” ./sha256/“db id”
  • 레이어의 로컬 저장 경로(/var/lib/docker/overlay2/{cache-id})

layer id : db id

  • 앞에가 layer id, 뒤에가 db id 이다.

parent id : db id

  • parent 파일은 부모레이어의 db id를 갖고 있음 (가장 밑바닥은 parent 없음)

  • sha256:”db-id”./sha256/“parent-id”

  • 311627f8702d2d97e63b916cad711aa9f962a1f20da5572d68b8f223ff051e73 가 최상위 부모 db -id 이다.

레이어 스택

  • layer id : 레이어 식별
  • db id : 레이어관리
  • cache-id : 레이어 저장

실제저장 위치

  • cache id 폴더이름
  • l 디렉터리에는 레이어 저장소(cache-id) 심볼릭 링크가 존재
